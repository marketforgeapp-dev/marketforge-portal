# Internal Lead Capture SOP

\# MARKETFORGE — LEAD CAPTURE SOP  
\#\# Internal operating procedure for entering and attributing booked jobs

\#\#\# Purpose  
This SOP defines exactly how MarketForge captures, attributes, and reports booked jobs for early customers like Masterflo Plumbing.

The goal is:  
\- clean data  
\- consistent attribution  
\- clear reporting  
\- minimal burden on the customer

This SOP should be followed every time lead/job data is entered into MarketForge.

\---

\#\# Core operating model  
This is a \*\*concierge lead capture workflow\*\*.

\#\#\# Customer responsibility  
Customer provides lightweight booked-job updates.

\#\#\# MarketForge responsibility  
MarketForge:  
\- enters the data  
\- assigns attribution  
\- normalizes service names  
\- tracks revenue  
\- keeps reporting consistent

\---

\#\# What gets entered into MarketForge  
Only enter \*\*booked jobs\*\* into the leads system.

Do NOT enter:  
\- general inquiries  
\- raw calls  
\- unqualified leads  
\- duplicate messages  
\- non-booked estimates  
\- spam

If a job is booked, it belongs in MarketForge.

\---

\#\# Required data for every entry  
Every job entered must include:

1\. Customer name  
2\. Service type  
3\. Revenue value  
4\. Date booked  
5\. Workspace / customer account  
6\. Attribution/source  
7\. Notes, if needed

If any field is missing, resolve it before finalizing entry unless the SOP explicitly allows “unknown.”

\---

\#\# Source hierarchy and attribution rules  
Attribution must follow this exact hierarchy.

\#\#\# Tier 1 — Explicit attribution  
Use this when the customer clearly indicates the source.

Examples:  
\- “Customer mentioned Facebook ad”  
\- “Came from Google Business post”  
\- “Called from emergency plumbing ad”

Rule:  
Assign directly to the matching MarketForge action/campaign.

\---

\#\#\# Tier 2 — Strong likely attribution  
Use this when the source was not explicitly stated but there is a strong connection.

Examples:  
\- job booked during the active window of a matching action  
\- service matches a currently running campaign very closely  
\- landing page / offer / CTA clearly aligns with one launched action

Rule:  
Assign to the most likely related action and mark internally as “likely influenced.”

\---

\#\#\# Tier 3 — Unknown / organic  
Use when there is not enough evidence.

Examples:  
\- customer source not provided  
\- multiple possible campaigns  
\- no clear source clue  
\- customer says “found you online” with no detail

Rule:  
Assign to “organic / unknown.”

Do not force attribution.

\---

\#\# Attribution decision rule  
Use this rule every time:

\#\#\# If explicit  
assign to exact action

\#\#\# If highly likely  
assign to likely influenced action

\#\#\# If unclear  
assign to organic / unknown

This rule is mandatory.

\---

\#\# Revenue rules  
\#\#\# Preferred  
Use actual booked job value.

\#\#\# If actual value is not available  
Use the best available estimate.

\#\#\# If estimate is used  
Mark it internally or note it so it can be corrected later if needed.

Never leave revenue blank if a reasonable value is available.

\---

\#\# Date rule  
Use the \*\*date the job was booked\*\*, not:  
\- date reported to MarketForge  
\- date completed  
\- date invoiced

If only completion date is known and booked date is unavailable, use the best known date and add a note.

\---

\#\# Service naming rules  
Use the customer’s natural service description first, then normalize if needed.

Examples:  
\- “water heater replacement”  
\- “emergency plumbing”  
\- “drain cleaning”  
\- “slab leak repair”

If the service wording is vague:  
\- map it to the nearest real service family  
\- add a note if uncertain

Do not invent a service category that does not exist in the business.

\---

\#\# Duplicate prevention rule  
Before entering a job, check whether it has already been entered.

Use:  
\- customer name  
\- service  
\- revenue  
\- booking date

If it appears to be the same booked job:  
\- update existing entry if needed  
\- do not create a second record

\---

\#\# Correction rule  
If the customer later sends:  
\- corrected revenue  
\- corrected source  
\- cancellation  
\- change in service type

Update the existing lead/job record rather than creating a new one.

Always preserve consistency in reporting.

\---

\#\# Entry frequency  
\#\#\# Minimum standard  
Enter updates once per week.

\#\#\# Recommended standard  
Enter updates twice per week:  
\- Tuesday  
\- Friday

Do not allow more than 7 days of backlog.

\---

\#\# Workflow for each update batch  
Follow this exact sequence:

\#\#\# Step 1 — Collect source update  
Review customer-provided update from:  
\- text  
\- email  
\- note  
\- screenshot  
\- call summary

\#\#\# Step 2 — Break into individual jobs  
Separate each booked job into its own entry candidate.

\#\#\# Step 3 — Validate required fields  
Confirm:  
\- customer name  
\- service  
\- revenue  
\- booking date  
\- source clue if available

\#\#\# Step 4 — Normalize service  
Map service wording to the correct service family if needed.

\#\#\# Step 5 — Attribute source  
Use the attribution hierarchy:  
\- explicit  
\- likely influenced  
\- unknown

\#\#\# Step 6 — Check duplicates  
Search existing entries before creating a new one.

\#\#\# Step 7 — Enter into MarketForge  
Create or update the lead/job record.

\#\#\# Step 8 — Final review  
Verify:  
\- correct workspace  
\- correct action attribution  
\- correct revenue  
\- correct booking date  
\- no duplicate entry

\---

\#\# Reporting categories to maintain  
Every entered job should ultimately support reporting in one of these buckets:

1\. Attributed to specific MarketForge action  
2\. Likely influenced by MarketForge action  
3\. Organic / unknown

This structure must remain consistent across all customers.

\---

\#\# Customer-facing reporting rule  
When reporting results back to the customer:

\#\#\# Use clear commercial language  
\- booked jobs  
\- revenue generated  
\- actions launched  
\- likely influenced revenue

\#\#\# Do NOT overstate attribution  
If attribution is uncertain, say:  
\- “likely influenced”  
\- not “directly caused”

Trust matters more than inflated claims.

\---

\#\# What success looks like  
A good lead capture process means:  
\- customer burden stays low  
\- MarketForge data stays clean  
\- reporting feels trustworthy  
\- campaign impact is visible  
\- future recommendations improve

\---

\#\# Internal quality standard  
Before any customer review or reporting update, confirm:

\- no duplicate jobs entered  
\- no obviously wrong attribution  
\- revenue values are reasonable  
\- service naming is clean  
\- unknowns are labeled honestly

\---

\#\# Final rule  
Do not optimize for perfect attribution.  
Optimize for:  
\- consistency  
\- honesty  
\- usefulness  
\- repeatability

That is the standard for founder-led v1.
